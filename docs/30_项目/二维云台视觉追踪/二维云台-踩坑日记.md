# 踩坑日记 — 二维云台视觉追踪调试实录

---

## 坑1：DMA 空闲中断数据错位 — 帧头在中间

### 现象

K230 发来的坐标偶尔严重偏移（目标在画面中心，读到的坐标却是 (0, 0) 或 (65535, 65535)）。云台突然向一个方向猛转。

### 排查过程

1. **串口助手抓原始数据**
   ```
   正常: AA 55 01 40 00 F0 BB ...
   异常: 00 F0 BB AA 55 01 40 ...  ← 帧头错位！
   ```

2. **根因分析**
   - `HAL_UART_Receive_DMA` 从缓冲区的 0 位置开始接收
   - 如果 DMA 重启时线上刚好在传输一帧的中间字节，帧头 (0xAA 0x55) 就会出现在缓冲区的非 0 位置
   - `Parse_K230_Black_Target()` 虽然做了滑动搜索（`for (i=0; i<len-6; i++)`），但缓冲区的 `k230_rx_count` 可能导致搜索范围不够

### 解决方案

```c
// 方案1: 缓冲区中滑动搜索帧头（代码已有，需确认搜索范围覆盖整个缓冲区）
for (uint16_t i = 0; i < len - 6; i++) {
    if (pBuffer[i]==0xAA && pBuffer[i+1]==0x55 && pBuffer[i+6]==0xBB) {
        // 找到有效帧
    }
}

// 方案2: 使用环形缓冲区，避免帧撕裂
// 双缓冲 + ping-pong 切换
```

### 经验教训

> DMA + IDLE 中断接收时，帧不一定从 buffer[0] 开始。**必须做滑动搜索帧头**，不能假设数据对齐。

## 坑2：电机指令串扰 — 两轴同时发指令导致电机乱动

### 现象

YAW 和 PITCH 同时控制时，PITCH 偶尔执行的是 YAW 的指令，电机乱转。

### 排查过程

1. **逻辑分析仪抓 USART1 TX**
   - 发现两条指令之间的间隔只有约 10us
   - 电机驱动器的 UART 接收缓冲区可能没来得及处理第一条指令，第二条就覆盖了

2. **根因**
   - `X_V2_Vel_Control` 内部用 `HAL_UART_Transmit_DMA` 异步发送 4 字节
   - 虽然代码中有 `while(huart1.gState != HAL_UART_STATE_READY)` 等待 DMA 完成
   - 但 DMA 完成 ≠ 电机驱动器完成接收处理
   - 电机驱动器的 UART RX 是中断驱动的，需要时间解析

### 解决方案

```c
// 在每个电机指令之间加足够的延时
X_V2_Vel_Control(MOTOR_YAW_ID, dir_yaw, ACC, vel_yaw, false);
while(huart1.gState != HAL_UART_STATE_READY); // 等DMA发完
HAL_Delay(5);  // ← 多等5ms，确保电机驱动器消化完毕

X_V2_Vel_Control(MOTOR_PITCH_ID, dir_pitch, ACC, vel_pitch, false);
while(huart1.gState != HAL_UART_STATE_READY);
HAL_Delay(5);
```

### 经验教训

> DMA 发送完成到对方真正"听懂"之间有延迟。**多设备共享同一 UART 总线时必须加指令间隔**

## 坑3：K230 坐标跳零 — 视觉模块丢目标后发全零

### 现象

目标移出视野后，K230 发送坐标 (0, 0)，云台猛甩到极限位置。

### 排查过程

1. 当目标丢失时，K230 没有"目标丢失"的标志位，而是直接发送 (0, 0)
2. `Parse_K230_Black_Target` 把 (0,0) 当作有效坐标
3. `error_x = 320 - 0 = 320px`，P 控制器算出大速度，云台追向角落

### 解决方案

```c
// 在解析中增加合理性检查
if (target_x < 10 && target_y < 10) {
    // 坐标在左上角极值区，可能是目标丢失
    // 停止追踪，保持当前位置
    return;
}

if (target_x > 630 || target_y > 470) {
    // 超出图像范围（640x480），无效坐标
    return;
}
```

### 经验教训

> **永远不要信任传感器发来的数据**。坐标 (0,0) 可能是"没找到目标"的默认值，必须做合理性校验。

## 坑4：上电归零 500ms 死等不够 — Flash 写入超时

### 现象

上电后第一次追踪，云台零点偏了大约 10~15度。断电重启后零点又变了。

### 排查过程

1. `X_V2_Reset_CurPos_To_Zero()` 指令触发电机内部写 Flash 存储零点
2. 数据手册标注 Flash 写入时间最长 **20ms**
3. 代码中用 `HAL_Delay(500)` 等待，理论上是够的
4. 但示波器抓取电机供电发现：归零指令后电机电流有约 200ms 的波动（编码器校准）
5. 实际 Flash 写入发生在编码器校准**完成之后**，而非收到指令时

### 解决方案

```c
// 方案1: 加大延时到 800ms
X_V2_Reset_CurPos_To_Zero(MOTOR_YAW_ID);
while(huart1.gState != HAL_UART_STATE_READY);
HAL_Delay(800);  // ← 从500ms加到800ms

// 方案2: 通过读状态寄存器确认完成（推荐，但需要电机固件支持）
// X_V2_Read_Sys_Params_Timed(MOTOR_YAW_ID, S_FLAG, 1000);
```

### 经验教训

> 电机驱动器内部的"指令→执行→完成"有时间差。**写 Flash 不是收到指令就开始的**，要等机械动作完成。加宽延时是最粗暴但有效的办法。

## 坑5：PID 参数调优 — KP 太大振荡，太小追不上

### 现象

KP=0.5 时：云台在目标周围高频振荡（抖得像帕金森）
KP=0.05 时：目标移动后云台慢悠悠转过去，追不上

### 排查过程

这是一个经典的 P 控制器参数整定问题：

```
速度 = 像素偏差 x KP

KP=0.5, 偏差=100px → 速度=50 → 冲过目标 → 过冲 → 反向 → 振荡
KP=0.05, 偏差=100px → 速度=5  → 太慢 → 目标跑远了
```

### 解决方案

```c
#define SPEED_KP_YAW     0.15f   // 最终收敛值
#define SPEED_KP_PITCH   0.15f
#define MAX_VEL_YAW      300.0f  // 加上速度限幅防止过冲
#define MAX_VEL_PITCH    200.0f
#define ACC_YAW          400     // 加速度平滑过渡
#define ACC_PITCH        400

// 死区 10px：防止到达目标后微小偏差导致的持续微调
if (fabs(error_x) <= 10.0f) {
    vel_yaw = 0.0f;
}
```

### 经验教训

> P 控制器整定口诀：**先调 KP 到不振荡的临界值，再加死区消除静态抖动，最后用速度限幅防过冲**。参数之间互相制约，需要系统性地调。

## 总结

| 排名 | 坑 | 教训 |
|------|-----|------|
| 1 | DMA 帧数据错位 | DMA+IDLE 接收必须滑动搜索帧头 |
| 2 | 电机指令串扰 | 共享 UART 总线，指令间加 5ms 间隔 |
| 3 | K230 丢目标发(0,0) | 永远校验传感器数据的合理性 |
| 4 | 归零 Flash 超时 | 等电机动作完成再算时间，加宽延时 |
| 5 | PID 参数整定 | 系统性调节：KP→死区→限幅→加速度 |
