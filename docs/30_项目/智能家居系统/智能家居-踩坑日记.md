## 坑1：ESP8266 AT 指令阻塞 UI — TouchGFX 卡死不响应

### 现象

按下 TouchGFX 上的"开电视"按钮后，整个屏幕卡住 2~3 秒，触控无响应。松手后恢复。

### 排查过程

1. **调用链分析**
   ```
   TouchGFX Task(UI线程) → opentv() → ESP8266_Broadcast()
   → ESP8266_Send_To_ID() → Send_AT_Command_Ext() → while循环等"SEND OK"
   ```

2. **根因**: `Send_AT_Command_Ext()` 内部有 `while` 循环轮询 `rx_buffer` 等 "SEND OK"，超时 1 秒。3 个 ESP32 节点 = 3 秒阻塞。
   - UI 线程是单线程的 TouchGFX Task
   - 阻塞期间无法处理触控事件 → 屏幕卡死

### 解决方案

**引入第二个 FreeRTOS 任务 + 消息队列**:

```c
// 1. 创建 WiFi 发送队列
osMessageQDef(wifiQueue, 16, uint32_t);
wifiCommandQueueHandle = osMessageCreate(osMessageQ(wifiQueue), NULL);

// 2. 创建独立 WiFi 发送任务
osThreadDef(wifiSendTask, StartWifiSendTask, osPriorityNormal, 0, 512);
osThreadCreate(osThread(wifiSendTask), NULL);

// 3. UI 线程只投递消息，瞬时返回
void Screen2View::opentv() {
    if (toggleButton1.getState()) {
        HAL_GPIO_WritePin(GPIOB, GPIO_PIN_0, GPIO_PIN_RESET);
        osMessagePut(wifiCommandQueueHandle, (uint32_t)"OPENTV_ON\n", 0);
        // ← 立即返回！不阻塞！
    }
}

// 4. WiFi 任务在后台慢慢发
void StartWifiSendTask(void const * argument) {
    for(;;) {
        evt = osMessageGet(wifiCommandQueueHandle, osWaitForever);
        ESP8266_Broadcast((char*)evt.value.p);  // 阻塞在这里不影响 UI
    }
}
```

### 经验教训

> **UI 线程绝不能阻塞！** 任何涉及 `while` 等待的操作（AT指令、I2C、SPI）都必须在独立任务中执行，通过消息队列与 UI 线程通信。

## 坑2：UART3 中断和 AT 应答抢 rx_buffer — OK 被 AI 指令冲掉

### 现象

`Send_AT_Command_Ext()` 偶尔超时返回 0，即使 ESP8266 实际已回复 OK。WiFi 初始化偶尔失败。

### 排查过程

1. **rx_buffer 共享问题**
   - UART3 中断 (`HAL_UART_RxCpltCallback`) 写入 `rx_buffer`
   - `Send_AT_Command_Ext()` 的 while 循环读取 `rx_buffer` 等 "OK"
   - 如果在等 OK 期间，AGX Orin 发来一帧 AI 指令(JSON)，中断会把 `rx_buffer` 覆盖
   - "OK" 被冲掉 → while 循环超时 → 返回失败

2. **时序分析**
   ```
   ESP8266:   AT+CIPSEND=0,10\r\n → OK\r\n >
   AGX Orin:  (同时) {"action":"open","device":"电视"}  ← 竞争！
   ```

### 解决方案

```c
// 发送 AT 指令前清空缓冲区并关中断保护
uint8_t Send_AT_Command_Ext(char *cmd, uint32_t timeout_ms, uint8_t wait_for_prompt) {
    __disable_irq();  // ← 原子操作
    rx_index = 0;
    memset(rx_buffer, 0, sizeof(rx_buffer));
    is_ok_received = 0;
    is_prompt_received = 0;
    __enable_irq();

    HAL_UART_Transmit(&huart3, (uint8_t*)cmd, strlen(cmd), 100);

    // while 循环等 OK 或 > 或超时
    ...
}
```

### 经验教训

> 共享缓冲区 + 中断写入 = 竞态条件。**读写缓冲区前关中断**是最小成本的原子操作。更好的方案是为 AT 应答和业务数据准备独立缓冲区。

## 坑3：UTF-8 中文匹配踩坑 — strstr 找不到 "空调"

### 现象

AGX Orin 发来的指令明明是 "打开空调"，但 `strstr(raw_data, "空调")` 总是返回 NULL。

### 排查过程

1. 在 Keil 中 `"空调"` 的编码取决于源文件编码
2. 如果 `.cpp` 文件是 UTF-8，而 AGX Orin 发的也是 UTF-8 → 匹配成功
3. 如果 `.cpp` 文件是 GB2312（Keil 默认），而 AGX Orin 发的是 UTF-8 → 编码不一致 → 匹配失败
4. Keil MDK 默认源文件编码是 ANSI/GB2312，不支持直接在字符串里写中文

### 解决方案

**用 UTF-8 字节序列替代中文字面量**:

```c
// 不能用: strstr(raw_data, "空调")
// 改用 UTF-8 字节序列:
// "空调" 的 UTF-8: E7 A9 BA E8 B0 83

if (strstr(raw_data, "\xE7\xA9\xBA\xE8\xB0\x83") != NULL) {
    // 匹配到"空调"
}

// 常用 UTF-8 字节序列速查:
// "开" = \xE5\xBC\x80
// "关" = \xE5\x85\xB3
// "电视" = \xE7\x94\xB5\xE8\xA7\x86
// "窗帘" = \xE7\xAA\x97\xE5\xB8\x98
// "监控" = \xE7\x9B\x91\xE6\x8E\xA7
// "制热" = \xE5\x88\xB6\xE7\x83\xAD
// "制冷" = \xE5\x88\xB6\xE5\x86\xB7
```

### 经验教训

> Keil 源文件编码 ≠ 运行时数据编码。**永远用十六进制字节序列做中文匹配**，不依赖源文件编码。

## 坑4：ESP8266 TCP Server 多连接时 ID 混乱

### 现象

发送 `AT+CIPSEND=0,10` 给 ID 0 的设备，实际收到指令的却是 ID 2 的设备。

### 排查过程

1. `AT+CIPMUX=1` 启用多连接后，ESP8266 给每个连入的客户端分配 Link ID (0~4)
2. 连接顺序决定 ID：第一个连入的 AGX Orin 可能是 ID 0，第二个 ESP32 是 ID 1...
3. 但 WiFi 连接不稳定时设备会断开重连，**ID 会重新分配**
4. 代码里硬编码 `for (i=0; i<3; i++)` 假设 ID 0,1,2 始终是三个 ESP32

### 解决方案

```c
// 方案1: 广播模式 — 所有设备收到同一条指令，由设备端根据内容判断是否执行
void ESP8266_Broadcast(char* payload) {
    for (uint8_t i = 0; i < 5; i++) {  // 尝试所有可能的 ID
        if (ESP8266_Send_To_ID(i, payload)) {
            // 发送成功（即使该 ID 实际不存在，也不影响）
        }
    }
}

// 方案2: 设备主动注册 — ESP32 连上后发送注册包
// "REG:TV\n" → STM32 记录: ID 2 = TV
// 更复杂但更可靠
```

### 经验教训

> ESP8266 多连接的 Link ID 是**动态分配的**，不能硬编码。广播模式虽然"粗暴"但最可靠。

## 坑5：Soft_Delay_ms 精度严重依赖编译器优化等级

### 现象

-O0 (无优化) 时 Wi-Fi 连接 100% 成功，-O2 优化后 `Send_AT_Command_Ext` 频繁超时。

### 排查过程

1. `Soft_Delay_ms()` 是忙等循环:
   ```c
   for (i = 0; i < ms; i++)
       for (j = 0; j < 25000; j++)
           __NOP();
   ```
2. -O2 优化下，编译器可能把内层循环优化掉（认为无副作用）
3. 即使没被完全优化掉，25000 次循环在 -O2 下的执行时间也完全不同

### 解决方案

```c
// 用 SysTick (HAL_Delay) 替代软延时
// 但注意: HAL_Delay 不能在中断中调用（依赖 SysTick 中断）
// 所以保留 Soft_Delay_ms 给 UART3 中断回调中的场景

// 或在 Send_AT_Command_Ext 中用 HAL_Delay 替代:
uint8_t Send_AT_Command_Ext(char *cmd, uint32_t timeout_ms, ...) {
    ...
    while (1) {
        if (wait_for_prompt && is_prompt_received) return 1;
        if (!wait_for_prompt && is_ok_received) return 1;
        if (wait_ms >= timeout_ms) return 0;
        
        HAL_Delay(10);  // ← 用 HAL_Delay 替代 Soft_Delay_ms(10)
        wait_ms += 10;
    }
}
```

### 经验教训

> 软延时(NOP 循环)的精度**完全依赖编译器优化等级**。能用硬件定时器就别用软延时。SysTick (`HAL_Delay`) 是更好的选择。

## 总结

| 排名 | 坑 | 教训 |
|------|-----|------|
| 1 | AT 指令阻塞 UI | UI 线程零阻塞，耗时操作用独立任务+队列 |
| 2 | rx_buffer 数据竞争 | 共享缓冲区操作关中断 |
| 3 | UTF-8 中文匹配 | 用 \x 十六进制字节序列，不依赖源文件编码 |
| 4 | TCP 多连接 ID 混乱 | Link ID 动态分配，用广播而非指定 ID |
| 5 | 软延时优化依赖 | 优先 HAL_Delay，避免 NOP 循环 |
