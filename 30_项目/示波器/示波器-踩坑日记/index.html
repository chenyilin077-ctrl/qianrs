    踩坑日记 — 数字示波器调试实录

    坑1：ETR 外部时钟模式选错 — 频率永远是 0

    现象

    TIM1 配置了 ETR，代码也启动了，但 __HAL_TIM_GET_COUNTER(&htim1) 始终返回 0。外部信号明明有 1kHz 的方波输入 PA12。

    排查过程

    1. CubeMX 配置回顾
    2. TIM1 Clock Source 选了 ETR2（External Trigger）

    3. 但代码中 sClockSourceConfig.ClockSource = TIM_CLOCKSOURCE_ETRMODE2

    4. 关键区别

    5. ETRMODE1：ETR 信号经过边沿检测后触发计数器+1 —— 这是我们需要的
    6. ETRMODE2：ETR 信号直接作为计数器时钟源 —— 每个上升沿计数一次

    7. CubeMX 里的 ETR2 对应代码里的 ETRMODE2

    8. 真正的问题

    9. 外部信号是 3.3V 方波
    10. PA12 配置为 GPIO_MODE_AF_PP（推挽复用输出）
    11. 但作为 ETR 输入时，引脚应该是输入模式，不是输出！
    12. GPIO_AF1_TIM1 是复用功能，但 GPIO_MODE_AF_PP 是输出模式

    解决方案

    // 修正前 (CubeMX 默认生成)
GPIO_InitStruct.Mode = GPIO_MODE_AF_PP;   // 推挽复用输出 ← 错误！

// 修正后
GPIO_InitStruct.Mode = GPIO_MODE_AF_OD;   // 开漏复用
// 或
GPIO_InitStruct.Mode = GPIO_MODE_AF_PP;   // 可以，但要确认 CubeMX
GPIO_InitStruct.Pull = GPIO_PULLUP;       // 加上拉

    但实际上关键是要确保 TIM1 的 ETR 配置不被别的外设干扰： - 确认 PA12 没有被其他外设（SPI、UART）复用 - 确认 __HAL_RCC_TIM1_CLK_ENABLE() 已执行

    经验教训

    ETR 模式下引脚 Mux 要仔细检查。GPIO_MODE_AF_PP 虽然可以工作但不够严谨——ETR 是输入功能，用 OD 或确认 CubeMX 自动选择 AF 模式。

    坑2：DMA 循环模式下的数据竞争 — 波形撕裂

    现象

    TouchGFX 显示的波形偶尔出现"撕裂"——前半段是上一帧的老数据，后半段是新数据。或者电压读数跳动。

    排查过程

    1. ADC DMA 是循环模式：hdma_adc1.Init.Mode = DMA_CIRCULAR
    2. DMA 持续写入 adc_buff[1024]，同时 TouchGFX 在 handleTickEvent() 中读取
    3. 没有同步机制！ DMA 写入和 CPU 读取是并发的
    4. 如果 CPU 刚好读到 DMA 正在写入的位置，就会读到半新半旧的数据

    解决方案

    // 方案1: 双缓冲 Ping-Pong
uint16_t adc_buff[2][1024];
volatile uint8_t active_buffer = 0;

// DMA半传输中断 → 切换缓冲区
void HAL_ADC_ConvHalfCpltCallback(ADC_HandleTypeDef* hadc) {
    active_buffer = 0;  // 前半段就绪
}
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef* hadc) {
    active_buffer = 1;  // 后半段就绪
}

// TouchGFX 读取 inactive 缓冲区
// 方案2 (当前采用的折中): 只读DMA写入位置之前的数据
uint16_t curr_ptr = 1024 - __HAL_DMA_GET_COUNTER(hadc1.DMA_Handle);
// 只绘制 curr_ptr 之前已完成的采样点

    经验教训

    DMA 循环模式 + CPU 读取 = 必须同步。最简单的是读 DMA 当前指针，只读取 DMA 已经写过的安全区域。

    坑3：峰值保持衰减速率调不好 — 电压要么卡死要么狂跳

    现象

    • 衰减太快（50mV/tick）：电压显示从 3.3V 骤降到 0V，中间完全看不到下降过程
    • 衰减太慢（1mV/tick）：输入信号已经断开，电压显示还停在 3.3V，要等很久才降下来
    • 不衰减：拔掉信号线后电压永远显示上次的峰值

    排查过程

    刷新率分析： - TouchGFX tick ≈ 60Hz (16.7ms) - handleTickEvent 每 3 帧执行一次 → 50ms 间隔 - 每 50ms 衰减一次

    衰减速率 vs 用户体验： 

    10mV/tick × 20 ticks/s = 200mV/s  ← 3.3V → 0V 需要 16.5秒 (太慢)
50mV/tick × 20 ticks/s = 1V/s     ← 3.3V → 0V 需要 3.3秒  (勉强)
100mV/tick × 20 ticks/s = 2V/s    ← 3.3V → 0V 需要 1.6秒  (合适但有点跳)

    解决方案

    static float v_max_held = 0.0f;

// 从1024点中找真正的最大值（而不是窗口中某个）
uint16_t local_max = find_max_in_buffer();

if (local_max > v_max_held) {
    v_max_held = local_max;           // 跟涨: 瞬间响应
} else {
    v_max_held -= 10.0f;              // 衰减: 200mV/s
    if (v_max_held < local_max) {
        v_max_held = local_max;       // 下限保护
    }
}

    最终收敛到 10mV/tick。在 50ms 刷新下 = 200mV/s 的衰减率，3.3V→0 约 16 秒，对慢变信号友好。对于快变信号，新的峰值会瞬间覆盖旧值。

    经验教训

    峰值保持的衰减速率是用户体验参数，需要根据刷新率反算。太快=看不到下降过程，太慢=卡死。10mV/tick 是一个好的起点。

    坑4：ETR级联的竞态 — 读到撕裂的48bit计数值

    现象

    频率读数偶尔跳到一个巨大的异常值（如 4294967295 Hz），然后下一帧恢复正常。

    排查过程

    1. 48bit 计数器由两次 32bit 读操作拼出： 
      uint32_t low_16  = __HAL_TIM_GET_COUNTER(&htim1);  // 读TIM1
uint32_t high_16 = __HAL_TIM_GET_COUNTER(&htim2);   // 读TIM2
    2. 如果在这两次读操作之间，TIM1 发生了溢出（从 65535 → 0），TIM2 会+1
    3. 场景：读 TIM1=65534，然后 TIM1溢出→TIM2+1，再读 TIM2=100
    4. 结果：拼出来 (100 << 16 | 65534) = 但是 correct 的应该是 (99 << 16 | 65534)
    5. 误差 = 65536 个脉冲

    解决方案

    // 双重读取 + 校验
uint32_t high1, high2, low;
do {
    high1 = __HAL_TIM_GET_COUNTER(&htim2);
    low   = __HAL_TIM_GET_COUNTER(&htim1);
    high2 = __HAL_TIM_GET_COUNTER(&htim2);
} while (high1 != high2);  // 如果读low期间TIM1溢出了，high会变化，重读

uint32_t current = (high1 << 16) | low;

    经验教训

    多寄存器拼合宽计数器时，必须处理竞态。双重读取 + 比较是最简单的软件解决方案。

    坑5：3点滑动平均引入相位延迟 — 高频方波变正弦波

    现象

    输入 50kHz 方波，屏幕显示的是类似正弦波的波形，上升沿完全消失。

    排查过程

    1. 采样率 200kHz，50kHz 信号每周期只有 4 个采样点
    2. 3点滑动平均 = 低通滤波器，截止频率 ≈ 采样率/6 ≈ 33kHz
    3. 50kHz 在截止频率之外，高频分量被滤掉
    4. 方波的谐波（150kHz, 250kHz...）全部被滤掉 → 只剩基频 → 变成正弦波

    解决方案

    // 高频时关闭滤波，或动态调整滤波窗口
if (final_frequency > 30000) {  // >30kHz, 关滤波
    // 原始值直接画
    dynamicGraph1.addDataPoint(raw_mv);
} else {
    // 低频保持3点平均
    dynamicGraph1.addDataPoint(avg_mv);
}

    另一个更好的方案：提高采样率。200kHz 对 50kHz 确实太低了（奈奎斯特要求至少 2×，实际需要 5~10×）。

    经验教训

    滤波器的截止频率必须考虑信号带宽。固定窗口的滑动平均 = 固定截止频率的低通滤波器，高频信号会被"平滑"变形。

    总结

    排名 坑 教训
    1 ETR时钟源配置 CubeMX ETR模式确认引脚方向和复用
    2 DMA循环数据竞争 只读DMA已完成的区域或用双缓冲
    3 峰值保持衰减调参 根据刷新率反算，10mV/tick起步
    4 48bit拼合竞态 双重读取+比较防溢出窗口
    5 滑动平均滤掉高频 根据信号频率动态调整滤波强度
